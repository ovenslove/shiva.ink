# 高级前端开发工程师（Vue）专业技能体系与评估标准（Shiva.ink）

> 目标：将“高级”从资历描述变为可量化的能力标准；用于招聘、晋升、培养、项目把关与持续改进。

## 1. 适用范围与分级定义

- **适用范围**：Vue 3（组合式 API）+ TypeScript + Vite + Pinia + Element Plus 的 Web 前端工程；可扩展至 SSR、微前端等更大规模场景。
- **分级定义**（能力级别，不等同职级）：
  - **L3（高级基线）**：可独立负责中等复杂度模块，能做性能与工程治理，能带人落地最佳实践。
  - **L4（专家）**：能做架构取舍与跨团队协作，能把复杂问题拆解为可交付方案，并建立机制避免复发。
  - **L5（领域负责人）**：能定义路线与标准，推动组织级治理（质量门禁、组件体系、平台化）。

## 2. 评分模型（可量化）

### 2.1 维度、权重与通过线

总分 100 分，采用“维度得分 × 权重”加权。

| 维度 | 权重 | 说明 |
|---|---:|---|
| Vue 3 组合式 API 与响应式原理 | 18 | 可写出可维护、可预测的响应式逻辑 |
| TypeScript 工程化集成 | 12 | 类型设计、边界收窄、可演进 |
| 状态管理（Pinia）与数据流 | 10 | 模块化、可测试、可追踪 |
| 组件设计与组件库开发 | 12 | API、可复用、可扩展、对齐 Element Plus |
| 性能优化与可观测 | 15 | 指标驱动、定位与治理、避免回退 |
| 测试体系（Vitest）与质量门禁 | 12 | 可测性设计、覆盖率与回归策略 |
| SSR/预渲染与 SEO（高级项） | 8 | 适用于未来演进与扩展 |
| 微前端与工程架构（高级项） | 8 | 适用于多团队/多应用协作 |
| 工程协作（Git/CI/发布） | 5 | 可复制的交付流程 |

**通过线建议**：

- **高级（L3）**：总分 ≥ 75，且 “Vue + 性能 + 测试” 三项合计 ≥ 40
- **专家（L4）**：总分 ≥ 85，且至少 2 个维度达到 90% 满分
- **领域负责人（L5）**：总分 ≥ 92，且具备可复用机制/平台化产出

### 2.2 单维度打分规则（0–5）

每个维度按 0–5 分打分，再换算为百分制：

- 0：不了解/无法独立完成
- 1：能在指导下完成，常出错
- 2：能独立完成基础任务，缺乏边界与稳定性
- 3：能在复杂场景下正确设计并落地，有复盘意识（高级基线）
- 4：能提出更优方案并指导他人，建立规范与工具（专家）
- 5：能沉淀为体系/平台，持续提升团队产出（负责人）

换算：维度百分制 = (得分 / 5) × 100，再乘权重。

## 3. 各维度掌握标准（对齐本项目栈）

### 3.1 Vue 3 组合式 API 与响应式原理（18%）

**L3 必须具备**

- 能解释并正确使用 `ref/reactive/computed/watch/watchEffect` 的边界与代价，避免无意义响应式与深度 watch。
- 能在组件生命周期内正确管理副作用（DOM 事件、定时器、第三方实例），确保可释放。
- 能拆分组件：识别“页面聚合 vs 可复用组件”，在保持可读性的同时降低耦合。
- 能处理复杂交互：过渡动画、移动端手势、滚动监听（如本项目 `App.vue` 的滚动处理）做到性能与可维护性兼顾。

**L4/L5 加分项**

- 产出并推广组合式逻辑复用策略（可测试的 composables、统一副作用管理）。
- 形成组件复杂度控制机制（静态分析、Review 清单、性能预算）。

### 3.2 TypeScript 工程化集成（12%）

**L3 必须具备**

- 能为组件 Props/Emits/Slots 建立清晰类型，并保持演进兼容。
- 能用 `unknown` + 类型收窄替代无边界 `any`，边界层类型设计清晰。
- 能识别并修复常见类型陷阱：联合类型缩窄、可选链误用、泛型约束缺失。

**L4/L5 加分项**

- 定义领域模型与类型边界（DTO/VO），让类型成为协作契约。
- 推动“类型失败即构建失败”的工程门禁与迁移策略。

### 3.3 状态管理（Pinia）与数据流（10%）

**L3 必须具备**

- 能拆分 store：状态、getter、action 职责清晰，避免 store 变成“巨型上帝对象”。
- 能在测试中使用 `@pinia/testing` 隔离依赖，并为 action 编写回归用例。
- 能设计可追踪数据流：请求态、空态、错误态，避免 UI 与数据状态纠缠。

**L4/L5 加分项**

- 建立状态管理规范：命名、模块边界、持久化策略、调试与日志策略。

### 3.4 组件设计与组件库开发（12%）

**L3 必须具备**

- 组件 API 可预测：Props/slots/emits 边界清晰；默认态/空态/错误态完整。
- 能与 Element Plus 对齐：避免重复造轮子，掌握二次封装策略（主题、样式、交互一致性）。
- 具备可测性：关键分支可通过单测触达；对第三方组件可 stub/mock。

**L4/L5 加分项**

- 设计并维护组件库规范：版本策略、变更说明、可视化文档（如 Storybook 或替代方案）。

### 3.5 性能优化与可观测（15%）

**L3 必须具备**

- 以指标驱动优化：能读懂并复现实验（Lighthouse、Performance trace、包体积分析）。
- 能在 Vue 场景定位性能问题：多余渲染、过度响应式、长任务、图片/视频资源策略。
- 能提出并落地“不会回退”的方案：性能预算、拆包与懒加载、缓存策略、降级策略。

**L4/L5 加分项**

- 建立性能治理机制：准入预算、自动化检测、线上指标看板与报警。

### 3.6 测试体系（Vitest）与质量门禁（12%）

**L3 必须具备**

- 能为关键组件与 store 编写单测/集成测试，覆盖关键行为与回归场景。
- 熟练使用 mock：路由、第三方库（本仓库已有 PhotoSwipe mock 示例）、网络层（如未来接入 CMS/接口）。
- 能推动“修 bug 必补测试”的团队习惯，并能在 PR 维度执行。

**L4/L5 加分项**

- 设计覆盖率策略与阈值门禁（全局 + 核心模块），并落地到 CI。

### 3.7 SSR/预渲染与 SEO（8%，高级项）

**L3 了解即可（不作为硬门槛）**

- 理解 SSR/SSG/ISR 的差异与适用场景（博客/作品展示通常适合 SSG/预渲染）。
- 能识别 SSR 兼容问题：`window/document` 使用、hydration mismatch、异步数据策略。

**L4/L5 加分项**

- 能给出从 Vite SPA 迁移到 SSG/SSR 的路线：阶段目标、风险与回滚方案。

### 3.8 微前端与工程架构（8%，高级项）

**L3 了解即可**

- 理解微前端的收益与成本（隔离、发布独立、样式/依赖冲突、性能开销）。
- 知道常见方案的关键点：路由集成、公共依赖治理、通信、样式隔离。

**L4/L5 加分项**

- 能为多团队协作提出可落地治理：依赖策略、构建/发布协议、观测与回滚。

### 3.9 工程协作（Git/CI/发布）（5%）

**L3 必须具备**

- 规范化提交（Conventional Commits）与可回滚发布意识。
- 能在 CI 中设计必要门禁：类型检查、测试、构建、产物校验。

## 4. 考核方式（可执行）

### 4.1 评估组合（建议）

- **代码评审（40%）**：以真实 PR 为证据，评估可维护性、边界与质量门禁意识。
- **实战任务（40%）**：完成 1 个与本项目贴合的任务（见 4.2），要求可运行、可测试、可解释取舍。
- **技术面试/答辩（20%）**：围绕“方案权衡、性能定位、测试设计、类型设计”进行追问。

### 4.2 Shiva.ink 场景实战题（示例题库）

- **题目 A（组件与性能）**：为相册/文章图片预览增加“渐进加载 + 低网降级”，要求：
  - 不引入明显包体积回退；
  - 能说明如何避免重复初始化第三方实例；
  - 至少提供 2 个测试用例（渲染分支 + 交互）。
- **题目 B（状态与类型）**：将媒体数据（文章/相册）抽象为可扩展的领域模型（类型 + store 设计），要求：
  - 能新增一种媒体类型而不大改现有代码；
  - store action 可测试；
  - 明确空态/错误态的数据流。
- **题目 C（工程化）**：为项目补齐“可度量的质量门禁”方案设计（不要求一定落地到代码），要求：
  - 说明如何在 CI 中强制执行；
  - 说明对开发体验的影响与迁移步骤；
  - 给出可回滚策略。

## 5. 学习路径（从可上手到可带队）

### 5.1 L2 → L3（6–8 周）

- 第 1–2 周：组合式 API 规范化（副作用管理、拆分组件、响应式成本）
- 第 3–4 周：TypeScript 类型设计（Props/Emits/领域模型、unknown 收窄）
- 第 5–6 周：测试与回归（Vitest、Pinia testing、mock 策略）
- 第 7–8 周：性能治理（Lighthouse、拆包、资源策略、性能预算）

输出要求（可验收）：

- 2 个可复用组件沉淀（API 稳定、可测）
- 1 次性能优化闭环（指标前后对比 + 复盘）
- 1 个回归用例补齐（真实 bug 复现 + 修复）

### 5.2 L3 → L4（8–12 周）

- 建立模块级规范（store 分层、组件边界、目录治理）
- 推动质量门禁落地（覆盖率阈值/构建门禁/性能预算）
- 设计并落地可复用机制（composables、组件封装、最佳实践清单）

### 5.3 L4 → L5（长期）

- 组织级治理：指标体系、平台化工具、跨团队协作协议
- 架构演进：SSG/SSR、微前端、可观测体系

## 6. 持续改进机制（与团队规范联动）

- **能力校准**：每季度进行一次维度评分校准（用真实 PR/事故复盘做证据），避免“凭感觉打分”。
- **指标闭环**：把性能与质量指标绑定到发布节奏：
  - 每次版本发布抽样 3 个核心页面性能对比；
  - 每次线上问题复盘必须给出“如何用测试/门禁避免复发”。
- **课程与题库维护**：题库每月更新 1 次，淘汰过时题、补充真实问题题。
